# Event Ledger

Two Spring Boot microservices implementing a financial event ledger with idempotency, out-of-order event handling, distributed tracing, and circuit-breaker resiliency.

---

## Architecture

```
 Client
   │
   ▼
┌─────────────────────────────┐
│        Event Gateway        │  :8080  (public-facing)
│  - Accepts & persists events│
│  - Idempotency check        │
│  - Circuit breaker          │
│  - Trace ID on all responses│
└────────────┬────────────────┘
             │  HTTP (W3C traceparent propagated)
             ▼
┌─────────────────────────────┐
│       Account Service       │  :8081  (internal)
│  - Applies transactions     │
│  - Derives balance via SUM  │
│  - Out-of-order safe        │
└─────────────────────────────┘
             │
             ▼
┌─────────────────────────────┐
│           Zipkin            │  :9411  (tracing UI)
└─────────────────────────────┘
```

Each service has its own H2 in-memory database. They communicate over HTTP. The gateway is the only entry point for clients.

### Option 1 — Maven (two terminals)

**Terminal 1 — Account Service**
```bash
cd account-service
mvn spring-boot:run
```

**Terminal 2 — Event Gateway**
```bash
cd event-gateway
mvn spring-boot:run
```

### Option 2 — Docker Compose (everything in one command)

```bash
cd event-ledger
docker-compose up --build
```

This starts Zipkin, Account Service, and Event Gateway together. Trace data is visible at **http://localhost:9411**.

## API Reference

### Event Gateway — port 8080

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/events` | Submit a new event. Returns 201 (new) or 200 (duplicate). |
| `GET` | `/events/{id}` | Fetch a single event by internal ID. |
| `GET` | `/events?account={accountId}` | List all events for an account, ordered by event timestamp. |
| `GET` | `/health` | Health check with database connectivity status. |
| `GET` | `/metrics` | Custom operation counters. |

**POST /events — request body**
```json
{
  "eventId": "evt-001",
  "accountId": "acct-alice",
  "type": "CREDIT",
  "amount": "500.00",
  "currency": "USD",
  "eventTimestamp": "2026-05-01T10:00:00Z",
  "metadata": { "source": "web" }
}
```

**POST /events — response body**
```json
{
  "id": 1,
  "eventId": "evt-001",
  "accountId": "acct-alice",
  "type": "CREDIT",
  "amount": 500.00,
  "currency": "USD",
  "eventTimestamp": "2026-05-01T10:00:00Z",
  "metadata": { "source": "web" },
  "receivedAt": "2026-05-30T10:00:00Z",
  "accountSyncStatus": "SYNCED"
}
```

`accountSyncStatus` is `SYNCED` when the Account Service accepted the transaction, or `PENDING` when it was unreachable (circuit breaker open) — the event is always saved regardless.

### Account Service — port 8081

| Method | Path | Description |
|--------|------|-------------|
| `POST` | `/accounts/{accountId}/transactions` | Apply a transaction (auto-creates account on first hit). |
| `GET` | `/accounts/{accountId}/balance` | Current balance (derived, never stored). |
| `GET` | `/accounts/{accountId}` | Account details with full transaction history. |
| `GET` | `/health` | Health check. |
| `GET` | `/metrics` | Custom operation counters. |

## Design Decisions

### Idempotency
The gateway stores a unique `eventId` with a database-level unique constraint. On every `POST /events`, it first checks if the `eventId` already exists. If it does, it returns the stored event with HTTP 200 — no duplicate processing, no duplicate forwarding to the Account Service.

### Out-of-Order Event Handling
Balance is never stored. It is always computed on demand as `SUM(CREDITs) - SUM(DEBITs)` via a JPQL aggregate query. This means events arriving out of chronological order are naturally handled — the balance is always derived from the full set of transactions regardless of insertion order. Transactions are returned ordered by `eventTimestamp` for display.

### Distributed Tracing
Every request gets a unique trace ID generated by OpenTelemetry. The Event Gateway propagates the trace to the Account Service via the W3C `traceparent` header (injected automatically by `RestTemplateBuilder`). The `X-Trace-Id` header is set on every response by a `HandlerInterceptor`. Traces are exported to Zipkin.

### Structured Logging
Both services log JSON via `logstash-logback-encoder`. Every log line includes `traceId` and `spanId` from the active OTel span, making it easy to correlate logs with traces in Zipkin.

### Circuit Breaker
`AccountServiceClient.applyTransaction()` is wrapped with a Resilience4j circuit breaker (`accountService` instance). If the Account Service fails on 50% of the last 5 calls, the circuit opens for 10 seconds. During this time the fallback returns `false`, the gateway logs a warning, and the event response carries `accountSyncStatus: PENDING`. The gateway always returns 201 — it never propagates a 5xx to the caller.

**Why I chose Circuit Breaker over the other two:**

**vs. Timeout + Retry with backoff**
Retry is optimistic — it assumes the failure is transient and the downstream will recover quickly. For an Account Service that is repeatedly failing (which is what the assignment says), retrying just makes things worse: every event submission holds a thread open longer, response times spike, and you're still hammering a service that's already struggling. Circuit breaker is pessimistic in the right way — it stops trying after a threshold and lets the system breathe.

**vs. Bulkhead**
Bulkhead protects the Gateway's own resources (thread pool, connection pool) by isolating the Account Service call into a separate pool. It's a defensive pattern about your service, not about the downstream. It doesn't help the downstream recover, and it doesn't give the client a meaningful response — it just limits the blast radius. For this assignment, the more impactful behavior to demonstrate is graceful degradation to the client, which circuit breaker handles directly.

**Conclusion**
The assignment mentions "return a meaningful error to the client" — that maps exactly to what circuit breaker does. The three states (CLOSED → OPEN → HALF-OPEN) give automatic recovery without manual intervention. And the PENDING sync status is a concrete, readable demonstration of graceful degradation: the event is captured, the client gets a 201, and the system self-heals when the Account Service comes back.
